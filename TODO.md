## 1. Structured output (JSON) для selectTopics и review

**Зачем.** Сейчас весь pipeline — prose-in-prose. `pipeline.ts:257` считает темы регуляркой `^\d+\.` (модель напишет «1)» — счётчик ноль, SPARSE_WEEK-guard ломается). `pipeline.ts:280` парсит вердикт через `startsWith('нужна переработка')`. `validator.ts:36` считает источники глифами 📎. Хрупко на каждом шве.

**Что делаем.** Ввести JSON-схему для `selectTopics`:
```
{ topics: [{ title, sourceUrl, sourceOrigin: 'hn'|'web', tier: 1|2|3 }],
  sparseWeek: bool }
```
И для `reviewPost`: `{ verdict: 'ok'|'minor'|'rework', notes: [...] }` (по аналогии с уже существующим `BossReviewOutput` в `boss-pipeline.ts:23-28`). Validator переезжает на структуру: `sections.length === topics.length`, каждый topic.sourceUrl присутствует в посте, и т.п.

Заодно добить дополнительные правила валидатора, которые сейчас лежат в персоне как декларации, но не enforced:
- 🐤 не должны стоять близко друг к другу (минимум N≈500 символов между двумя цыплёнками — иначе дисклеймер и приветствие дублируют эмодзи). Канонически: один в приветствии, один в финальной подписи, тело между.
- Ровно один disclaimer-блок «Я МАЛЕНЬКИЙ БОТ, Я ТОЛЬКО УЧУСЬ» в самом начале.
- Финальная подпись на отдельной строке для weekly-режима (для stream-режима — отдельный набор правил, см. #8).

**Эффект.** Убивает целый класс багов с парсингом. Разблокирует #2 и #4.

**Файлы:** `src/lib/pipeline.ts`, `src/lib/validator.ts`, `src/config/kesha-reviewer.txt`, `src/lib/__tests__/pipeline.test.ts`, `src/lib/__tests__/validator.test.ts`.

## 2. Анти-галлюцинация ссылок

**Зачем.** Сейчас НИЧТО в pipeline не проверяет, что URL в посте реальные. `kesha-persona.txt:104` явно велит модели не валидировать ссылки («Trust all sources»). `kesha-reviewer.txt:31` тоже не проверяет. `validator.ts:36` считает только символы 📎. Один выдуманный URL уходит в @psyreq — удар по доверию.

**Что делаем.** После `generatePost` извлекать все URL и `@handles` из финального текста, сверять с set'ом из `hnContext` + `webContext`. На mismatch — `fixPost` со списком конкретных выдуманных ссылок. Опционально HEAD-запрос для top-level проверки 200.

**Эффект.** Единственная реальная защита от hallucination в news-канале. После #1 чище (есть `topic.sourceUrl`), но не блокируется им.

**Файлы:** `src/lib/pipeline.ts`, новый `src/lib/url-checker.ts`, `src/lib/__tests__/`.

## 3. Расширение источников + кэш

**Зачем.** `sources.json:12-21` определяет `priority_sources` (Anthropic blog, OpenAI blog, TechCrunch, Habr) — поле НИГДЕ не читается в коде, мёртвая конфигурация. `fetchHackerNewsContext` делает один запрос в `shir-man.com/api/feed` без User-Agent, без retry, 8s timeout. При фейле fallback в `web_search` — главный источник галлюцинаций.

**Что делаем.** Переименовать `hackernews.ts` → `sources.ts`, добавить параллельный pull RSS Anthropic / OpenAI / Google AI / TechCrunch + текущий HN-feed. Дедуп по URL внутри одной выборки. Кэш в Netlify Blob на 1-2 часа. `priority_sources` оживает.

**Эффект.** Убивает SPOF на shir-man, даёт первоисточники. Расширяет белый список URL для проверки в #2.

**Файлы:** `src/lib/hackernews.ts` → `src/lib/sources.ts`, `src/config/sources.json`, `src/lib/pipeline.ts`.

## 4. Дедуп по URL/сущностям вместо blob'а текста

**Зачем.** `kesha-post-background.mts:107` хранит последние 4 `selectedTopics` как сырой prose-блок. `pipeline.ts:117-122` клеит их в system prompt с просьбой «не повторяй эти строки». Тот же релиз с другой формулировкой проскочит как новый, реальное продолжение истории блокируется как дубль.

**Что делаем.** Завести нормализованную таблицу в blob: `{ url, entities: ['Anthropic', 'Claude 4.7'], publishedAt, postId }`. `selectTopics` получает structured signal: «URL уже публиковался — пропусти», «эта модель упоминалась 2 недели назад — пометь как development, не как анонс».

**Эффект.** Настоящий entity-level антидубль. База для #5 и #7.

**Зависит от:** #1 (нужны structured topics с url/entities, чтобы было что хранить).

**Файлы:** `src/lib/pipeline.ts`, `netlify/functions/kesha-post-background.mts`, новый `src/lib/memory.ts`.

## 5. Сквозные сюжеты + self-callbacks

**Зачем.** `published-topics` сейчас используется только чтобы НЕ повторяться. Память можно использовать положительно: «3 недели назад гадал, выкатит ли Anthropic X — выкатили / не выкатили / отозвали». Канал начинает ощущаться как серия, а не фид. Под персону «стажёра, который учится» это идеальный ход.

**Что делаем.** В `generatePost` подмешивать релевантные прошлые упоминания тех же entities из storage от #4. В `kesha-persona.txt` добавить опциональный паттерн «callback to N weeks ago» как ещё один штрих после вывода (по аналогии с уже описанными опциональными ходами в персоне, строки 67-71).

**Эффект.** Превращает дайджест в канал с памятью. Самый «характерный» сдвиг.

**Зависит от:** #4 (нужна нормализованная история с entities).

**Файлы:** `src/lib/pipeline.ts`, `src/config/kesha-persona.txt`.

## 6. Реакция на комменты (только Степан)

**Зачем.** Сейчас публикация это памятник: пост ушёл, тред живёт без бота. У Степана могут быть реплики «расширь по теме X», «сравни с тем, что было в прошлом году», «переведи на простой язык» — Кеша может отвечать в персоне. Плюс этот канал ввода — основа для #7 (Степан как судья качества постов).

**Что делаем.** Webhook на reply'и в треде последнего поста. Hard auth-check: реагирует ТОЛЬКО на `from.id === STEPAN_TELEGRAM_ID` (никаких других пользователей, никаких exception'ов). Маленький набор интентов через парсер по аналогии с `boss-command-parser.ts`. Rate-limit. Контекст ответа = текст поста + реплика Степана.

**Эффект.** Жизнь после публикации. Character проявляется вне жёсткого формата дайджеста. Плюс открывает канал для editorial feedback в #7.

**Зависит от:** ничего, независимый.

**Файлы:** новый `netlify/functions/kesha-comment-webhook.mts`, новый `src/lib/comment-handler.ts`.

## 7. Editorial-rating loop (Степан как судья) + (v2) аудиторные метрики

**Зачем.** «Стажёр учится» сейчас только в смысле антидубля. Реального ground truth нет — нужна петля «что зашло, что нет», чтобы стиль реально эволюционировал из недели в неделю.

**Почему НЕ автоматические метрики канала (как было в первой версии).** Telegram Bot API не отдаёт views ботам вообще. Reactions и комменты доступны, но только через webhook (`message_reaction_count`), не polling — нужна отдельная инфраструктура с привязанной discussion-group и `setWebhook(allowed_updates)`. Плюс reactions всегда коррелируют с темой (релиз Anthropic vs мелкий vibe-coding), не со стилем — без normalization научимся «писать про большие новости», что и так делает топик-селектор. Reward hacking (модель учится клик-бейту) для образа Кеши прямо off-character.

**Что делаем (v1 — Степан-как-судья).** Поверх инфраструктуры #6 добавить новый интент в comment-handler: `/kesha rate <1-5> [комментарий]` (или просто реплай к посту с заданной командой). Рейтинг + заметка пишутся в memory-запись поста (storage от #4). В `generatePost` подмешиваем топ-3 высоко оценённых и низ-3 низко оценённых поста — но НЕ целиком, а как style-examples отдельных кусков (интро, секция, вывод) — благодаря structured output из #1 эти куски отделимы и можно сравнивать «интро A vs интро B» при разных темах:

```
УДАЧНЫЕ ИНТРО (по оценке Степана):
[3 примера + краткая заметка «живо», «честно про процесс»]

НЕУДАЧНЫЕ ИНТРО:
[3 примера + заметка «слишком сухо», «штамп»]

Опирайся на стиль первых, избегай паттернов вторых. Формулировки не копируй.
```

Аналогично для выводов и секций. Никакой webhook'и на reactions, никакой возни с discussion-group.

**Что делаем (v2 — аудиторные метрики, опционально позже).** Если захочется второго слоя ground truth: webhook на `message_reaction_count`, snapshot через 48-72ч (`settled: true`), z-score нормировка по последним N постам, инъекция как ВТОРОЙ блок примеров с пометкой «по реакциям аудитории». Editorial Степана остаётся приоритетным сигналом — он может override'нуть «реакций много, но плохо».

**Эффект.** Настоящий learning loop с editorial ground truth. Идеально под легенду — «стажёр учится у руководителя», не «стажёр оптимизирует под лайки».

**Зависит от:** #1 (отделимые куски — интро/секция/вывод как самостоятельные единицы), #4 (storage с post_id), #6 (канал ввода рейтинга от Степана).

**Файлы:** `src/lib/comment-handler.ts` (новый интент), `src/lib/memory.ts` (поле `editorialRating`), `src/lib/pipeline.ts` (инъекция примеров в `generatePost`). Для v2 (если делаем): новый `netlify/functions/kesha-telegram-webhook.mts` и новый `kesha-metrics-snapshot-background.mts`.

## 8. File-input pipeline: стрим/митинг-ноутс → пост

**Зачем.** Сейчас входов два: weekly cron и `boss-command` (raw text, лимит 3500 символов в `pipeline.json:48`). Транскрипции стримов и митинг-ноутс на 10-50К символов туда не помещаются. Степан проводит стрим раз в месяц — нужен способ превратить запись в пост без ручной нарезки.

**Что делаем.** Расширить `kesha-boss-background.mts`: ловить апдейты с `document`, у которого `mime_type` = `text/markdown` или `text/plain`. Скачивать файл через Telegram `getFile` + fetch. Прогонять через новый `stream-pipeline.ts`:
1. **Extract** — извлечь 2-3 ключевых темы (deep) + 2-3 минорных (для бандла). Промпт в новом `src/config/kesha-stream-extract.txt`.
2. **Generate** — написать пост в формате стрим-рекапа (см. эталон ниже). Stream-specific persona override: подпись «Учусь. 🫡» вместо «Ваш стажер-Кеша», нет `📎` источников (источник = сам стрим), структура «2 deep + 1 бандл», более «робкий профессионал», менее «слэнговый стажёр».
3. **Review + validate** — отдельный `validateStreamPost` (без проверки `📎`, без проверки точной финальной подписи), reviewer с теми же поправками.

Auth — тот же `allowed_user_ids` из `pipeline.json:47`. Превью-флоу как в текущем boss-command (показать черновик в личке, дождаться подтверждения, потом постить в канал).

**Эталон формата.** `src/config/kesha-stream-example.txt` (уже в репо). Используется как few-shot пример в `stream-pipeline.ts` на этапе generate. Отличия от weekly-дайджеста, которые нужно отразить в stream-persona и stream-validator:
- Нет `📎` source-маркеров
- Структура: 2 темы развёрнуто (4-6 предложений каждая) + 1 бандл «коротко про остальное» с 3 минорными
- Интро короче (3 предложения), без банлиста weekly-дайджеста
- Закрытие — мягкое приглашение поправить в комментах вместо жёсткой подписи

**Зависит от:** ничего жёсткого. Чище после #1 (structured highlights extraction даёт JSON `{deep: [...], minor: [...]}`), но не блокировано — можно сделать prose-out как сейчас в weekly.

**Файлы:** `netlify/functions/kesha-boss-background.mts`, новый `src/lib/stream-pipeline.ts`, новый `src/config/kesha-stream-extract.txt`, новый `src/config/kesha-stream-persona.txt` (или флаг режима в существующей персоне), `src/lib/telegram.ts` (хелпер `downloadDocument`), `src/lib/validator.ts` (новый `validateStreamPost`), `src/config/pipeline.json` (новая секция `stream`).

## Логика порядка

- **#1 первым** — фундамент. Structured topics нужны для #2 и #4. Без него #2 работает на регэкспах поверх прозы, потом всё равно переписывать. И для #7 нужна отделимость интро/секций/вывода.
- **#2 вторым** — самый высокий risk/effort ratio. Credibility-фикс на день работы.
- **#3 третьим** — независимый, но даёт больше URL для белого списка #2 и убивает SPOF до того, как мы строим память поверх данных.
- **#4 → #5** — два уровня памяти канала. Сначала storage, потом positive use (callbacks).
- **#6 шестым** — независимая фича сама по себе («жизнь после публикации»), но одновременно строит канал ввода editorial-фидбека для #7. Удобнее делать ДО #7, потому что #7 без этого канала пришлось бы изобретать отдельный способ скармливать рейтинги.
- **#7 последним** — нужен #1 (отделимые куски), #4 (storage), #6 (канал ввода). Сводит всё вместе в learning loop.
- **#8 — параллельный трек.** Независим от 1-7, может делаться в любой момент. Чище после #1, но не блокирован. Эталонный пост (`kesha-stream-example.txt`) уже в репо, можно использовать как few-shot.
