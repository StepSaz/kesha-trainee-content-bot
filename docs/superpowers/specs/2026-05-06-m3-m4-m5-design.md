# Kesha Bot — M3/M4/M5 Design Spec

**Date:** 2026-05-06
**Status:** Approved

---

## Scope

Three milestones building on the M1+M2 foundation:

- **M3** — Normalized entity memory replacing prose-blob `published-topics`
- **M4** — Self-callbacks: inject past entity mentions into post generation
- **M5** — Telegram bot digest trigger + Stepan-only comment reactions

---

## M3: Normalized Memory (`src/lib/memory.ts`)

### Problem

`kesha-post-background.mts` stores the last 4 published posts as raw prose strings via `formatSelectedTopics()`. `pipeline.ts` passes these to `selectTopics` with a vague "don't repeat these" instruction. A topic with different wording but the same subject passes through as new.

### Design

New `src/lib/memory.ts` with three exports:

```typescript
interface MemoryEntry {
  url: string;        // topic.sourceUrl (empty string for migrated entries)
  title: string;      // topic.title
  publishedAt: string; // ISO datetime
  postId: number | null; // Telegram message_id, null if unknown
}

loadMemory(): Promise<MemoryEntry[]>
appendMemory(entries: MemoryEntry[]): Promise<void>
findCallbacks(newTopics: SelectedTopic[], memory: MemoryEntry[]): MemoryEntry[]
```

**Blob key:** `memory-v1`
**Max entries:** 40 (5 topics × 8 weeks, FIFO trim in `appendMemory`)

### One-time migration

`loadMemory()` checks for `memory-v1` first. If not found, reads `published-topics` blob and parses it with:

```
/\d+\.\s+\[T\d\]\s+(.+?)\s+\((?:hn|web)\)/g
```

Creates `MemoryEntry` per title with `url = ''`, `publishedAt ≈ now`, `postId = null`. Saves as `memory-v1`. On next `appendMemory()` the migration path is never hit again. `published-topics` blob is left to expire naturally.

### PipelineOptions change

```typescript
// before
publishedTopics?: string[];

// after
memoryEntries?: MemoryEntry[];
```

`selectTopics()` receives `memoryEntries` and builds a dedup block: URLs and titles seen in the last 8 weeks. The existing prose-block instruction is replaced with a structured signal per entry.

### kesha-post-background.mts changes

- Replace `store.get('published-topics')` with `loadMemory()`
- Replace `store.setJSON('published-topics', ...)` with `appendMemory(newEntries)`
- Pass `postId` from `sendResult.messageId` when appending

## M4: Self-Callbacks

### Problem

`published-topics` memory is used only negatively (don't repeat). The same history can be used positively: if Kesha mentioned a model 4 weeks ago and it shipped, she can reference the earlier mention.

### Design

`findCallbacks(newTopics, memory)`:
- Filters memory entries from **2–8 weeks ago** (< 2 weeks = dedup zone, > 8 weeks = stale)
- Matches by keyword: for each new topic title, splits into words ≥ 5 chars (filters prepositions/articles), checks if any memory entry title contains at least one matching word
- Returns up to 3 matching entries

In `generatePost()`, if `findCallbacks` returns results, injects an optional block into the system prompt:

```
CALLBACK CONTEXT (используй по желанию — только если органично):
- "GPT-5 Flash" упоминался 3 недели назад → можно сослаться как на развитие истории
- "DeepSeek V3" упоминался 6 недель назад
```

In `kesha-persona.txt`, one new paragraph added to the optional stylistic moves section: callback pattern "а помнишь, N недель назад мы гадали..." — framed as a trainee noticing a follow-up story.

## M5: Bot Digest Trigger + Comment Reactions

### Architecture

Both flows added to `kesha-boss-background.mts` (same Telegram webhook endpoint). New routing:

```
callback_query                                    → handleCallbackQuery
message, chat.type === 'private'                  → existing boss commands + /digest
message, reply_to_message, from.id === STEPAN_ID  → handleCommentReply
```

`STEPAN_ID = 352830345` (already in `src/config/pipeline.json:boss_command.allowed_user_ids[0]`).

### /digest command flow

1. Stepan sends `/digest` in bot DM
2. Bot replies immediately: "⏳ генерирую... (~60-90 сек)"
3. Loads `memoryEntries` and `previousIntros` from blobs
4. Calls `generatePipelinePost({ memoryEntries, previousIntros })`
5. Stores full result in `pending-post` blob
6. Sends post text to Stepan with inline keyboard:
   - `[🧪 В тест]` → callback_data: `approve_test`
   - `[📢 В прод]` → callback_data: `approve_prod`
   - `[❌ Отмена]` → callback_data: `cancel_post`

### Callback query handler

- `approve_test` / `approve_prod`: reads `pending-post`, calls `sendToChannel(post, chatId)`, calls `appendMemory()` + updates `previous-intros`, deletes `pending-post`, edits message to "✅ Отправлено"
- `cancel_post`: deletes `pending-post`, edits message to "❌ Отменено"

### Comment reaction flow

Trigger: non-DM message with `reply_to_message` from `from.id === STEPAN_ID`.

**Rate limit:** max 3 reactions per post. Counter stored in `comment-reactions-{reply_to_message_id}` blob.

**Intent parsing** (keyword-based, same pattern as `boss-command-parser.ts`):
- `расширь` / `подробнее` / `больше` → `expand`
- `объясни` / `что значит` → `explain`
- `сравни` / `vs` → `compare`
- anything else → `freeform`

**Generation:** single `claude-haiku` call with:
- System: Kesha persona (trainee voice, no em-dashes, no markdown)
- Context: `reply_to_message.text` (the post) + intent + Stepan's message
- Max tokens: 300

Responds via `sendMessage` with `reply_to_message_id` set (replies in thread).
